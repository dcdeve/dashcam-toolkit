## R-001 — Files Hash Algorithm

**Decision**: SHA-256 over a sorted, newline-delimited list of `"<basename>:<fileSize>"` strings, truncated to 16 hex chars for storage efficiency.

**Rationale**:
- Uses only `fs.readdir` + `fs.stat` — no file content reads (matches FR-007 and SC-002 < 2 s budget).
- Sorting ensures the hash is order-independent (filesystem readdir order varies across OS and FS types).
- `node:crypto` `createHash('sha256')` is built-in — no new dependency.
- 16 hex chars (64-bit prefix) is astronomically collision-resistant for the domain (< 10,000 video files per folder).

**Algorithm**:
```
entries = fs.readdir(dir) recursively (video extensions only, same as listVideoFiles)
lines   = entries.map(e => `${basename(e)}:${stat(e).size}`).sort()
hash    = sha256(lines.join('\n')).slice(0, 32)   // 128-bit prefix
```

**Alternatives considered**:
- MD5: Avoided due to theoretical collision risk; no performance advantage over SHA-256 truncated.
- mtime-based: Rejected — mtime is not reliable across copy operations and network volumes.
- File content hash: Rejected — too slow for SC-002.

## R-002 — Trip-to-Folder Assignment Strategy

**Decision**: Query-time path-prefix matching. No FK added to the `trips` table.

**Rationale**:
- Trips are assigned to the folder whose path is the longest prefix of the first clip's path (FR-011).
- At query time: for a given folder path `P`, retrieve trips whose first clip satisfies `clip.path LIKE 'P%'`.
- "Longest prefix" disambiguation at the DB level: when listing trips for folder `P`, exclude trips already claimed by a longer-prefix folder `Q` where `Q LIKE 'P%' AND Q != P`. In practice, implemented as: for each trip, the canonical folder is `MAX(folder.path) WHERE clip.path LIKE folder.path || '%'` — i.e., longest matching folder wins.
- Adding a `folderId` FK to trips would require updating all trips on every rescan and complicates the edge case of nested folders. The query-time approach handles this correctly with a single SQL expression.

**Performance**: With an index on `clips.path`, `LIKE 'P%'` scans at most O(clips in folder). For SC-003 (< 300 ms navigation), this is acceptable for libraries up to ~50,000 clips.

**Alternatives considered**:
- `folderId` FK on `trips`: Simpler queries, but requires scan-time assignment and re-assignment on nested folder changes. Rejected for YAGNI.
- `folderId` FK on `clips`: Even more invasive. Rejected.

## R-003 — Scanner → Folder Record Lifecycle

**Decision**: `scanner.scan()` calls `folders.upsertAfterScan(dir, clips)` at the end of the `importing` phase, before `grouping`. The `folders` module computes the hash from the already-discovered `files[]` list (passed in) to avoid a second directory walk.

**Rationale**:
- Scanner already has the video file list after the listing phase — reusing it avoids a redundant `readdir` walk.
- `upsertAfterScan` receives: `dir: string`, `files: Array<{name: string, size: number}>`.
- The scanner module imports from `src/interfaces/folders.ts` (interface only), keeping dependency direction clean.

**Hook point in `scanner.ts`** (between `importing` and `grouping` phases):
```typescript
await folders.upsertAfterScan(dir, filesWithSizes);
```

**Alternatives considered**:
- Post-scan hook in IPC handler: Would mean the folder record is only created when scanning via the UI, not the CLI. Rejected (breaks Dual-Surface Parity).
- Separate CLI/IPC step: Redundant roundtrip. Rejected.

## R-004 — Rescan Flow: ImportDrawer Integration

**Decision**: Add `openRescanDrawer(folderPath: string)` to the Zustand store. ImportDrawer reads a new `importRescanMode: boolean` flag from the store; when true, the drawer mounts directly at the `confirm` step (skipping `detecting`) and the directory-picker UI is hidden.

**Rationale**:
- The spec requires the drawer to open at step 2 (confirm/preview) with directory selection bypassed (FR-005, clarification 2026-03-24).
- ImportDrawer already uses an `importInitialDir` field in the store (step 1 pre-population). Rescan mode reuses this field for the path and adds a `importRescanMode` boolean to skip `detecting → confirm` transition.
- No new drawer component — reuse keeps the surface-area small (Principle VI).

**State machine delta**:
```
Normal:   idle → detecting → confirm → scanning → done|error
Rescan:   idle → confirm (immediate) → scanning → done|error
```

**Alternatives considered**:
- New `RescanDrawer` component: Duplicates ~80% of ImportDrawer. Rejected.
- URL/query-param routing: No router in this app. Rejected.

## R-005 — Folder Delete Semantics

**Decision**: `folders.remove(folderId)` deletes the `scanned_folders` record and all `trips` that belong exclusively to that folder (via `tripsModule.remove(tripId)` for each). Clips are deleted via cascade or explicit DELETE in the same transaction. Files on disk are not touched.

**Rationale**:
- Spec clarification (2026-03-24): "Deleting a folder removes the folder record and its associated trips from the Library; the actual files on disk are not affected."
- Uses existing `tripsModule` interface for trip removal to keep DB writes in the trips module.
- Wrapped in a SQLite transaction to ensure atomicity.

## R-006 — Folder Sort Control

**Decision**: Sort state lives in the Zustand store (`folderSortOption` field). Sort is applied client-side on the `UIFolder[]` array in the Library component using a `useMemo`. No DB-level sorting needed.

**Options**:
| Key | Label | Sort expression |
|-----|-------|-----------------|
| `health-stale-first` | Health (stale first) — **default** | `health === 'stale' ? 0 : health === 'unknown' ? 1 : 2`, then `lastScanAt DESC` |
| `name-asc` | Name (A–Z) | `displayName.localeCompare()` |
| `last-scanned-desc` | Last scanned (newest) | `lastScanAt DESC` |
| `trip-count-desc` | Trip count | `tripCount DESC` |

## R-007 — Health Check Execution Model

**Decision**: Health checks run in the Electron main process via a dedicated IPC handler (`FOLDERS.HEALTH_CHECK`). The renderer dispatches one IPC call per folder concurrently (via `Promise.all`), and updates the store as each result arrives.

**Rationale**:
- Filesystem access is main-process only in Electron (renderer has no `fs` module).
- Concurrent IPC calls per folder allow partial rendering (cards show `checking` → resolve individually).
- SC-002 (< 2 s/folder) is achievable: `readdir` + `stat` for 500 video files takes ~50–200 ms on local storage.

**Unknown / inaccessible path handling**:
- If `fs.access(dir, R_OK)` throws → return `{ status: 'unknown' }`.

## R-008 — New IPC Channels

| Channel | Direction | Payload | Response |
|---------|-----------|---------|----------|
| `folders:list` | renderer→main | `{}` | `ScannedFolder[]` |
| `folders:healthCheck` | renderer→main | `{ folderId: string }` | `{ status: FolderHealthStatus }` |
| `folders:remove` | renderer→main | `{ folderId: string }` | `{ ok: true }` |

No `folders:get` channel needed — `folders:list` returns all data; detail view filters client-side.

## R-009 — Existing `folderPath` in UITrip

`UITrip.folderPath` is currently derived as `firstClip.path.replace(/[/\\][^/\\]+$/, '')` (parent directory of first clip). For the folder detail view, we need to match trips to a `ScannedFolder.path` which is the scan root, not necessarily the immediate parent. Solution: filter `UITrip` list by `uiTrip.folderPath.startsWith(scannedFolder.path)` — this correctly handles recursive subdirectories under the scan root.
